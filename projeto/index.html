Padrão Strategy

O padrão Strategy sugere que você pegue uma classe que faz algo específico em diversas maneiras diferentes e extraia todos esses algoritmos para classes separadas chamadas estratégias.

A classe original, chamada contexto, deve ter um campo para armazenar uma referência para um dessas estratégias. O contexto delega o trabalho para um objeto estratégia ao invés de executá-lo por conta própria.

O contexto não é responsável por selecionar um algoritmo apropriado para o trabalho. Ao invés disso, o cliente passa a estratégia desejada para o contexto. Na verdade, o contexto não sabe muito sobre as estratégias. Ele trabalha com todas elas através de uma interface genérica, que somente expõe um único método para acionar o algoritmo encapsulado dentro da estratégia selecionada.

Desta forma o contexto se torna independente das estratégias concretas, então você pode adicionar novos algoritmos ou modificar os existentes sem modificar o código do contexto ou outras estratégias.

💡 Ideia principal: separar regras em classes diferentes.

📌 Enunciado do Problema

Uma aplicação precisa simular o deslocamento de um usuário até o aeroporto. Esse deslocamento pode ser realizado utilizando diferentes meios de transporte, como carro, ônibus ou bicicleta.

Na implementação inicial, a escolha do meio de transporte é feita por meio de estruturas condicionais (if/else), onde cada opção executa um comportamento específico.

Entretanto, com o crescimento do sistema, novos meios de transporte podem ser adicionados, como táxi, metrô ou aplicativo de carona, tornando o código cada vez mais complexo, difícil de manter e pouco flexível.

Diante desse cenário, surge a necessidade de aplicar um padrão de projeto que permita encapsular esses diferentes comportamentos e possibilitar a troca dinâmica do algoritmo de execução sem alterar a estrutura principal do sistema.

Assim, o desafio consiste em refatorar a solução utilizando o padrão Strategy, de forma a eliminar estruturas condicionais extensas e promover maior organização, flexibilidade e facilidade de manutenção no código.

❌ Código SEM Strategy

class Transporte {

    public void irParaAeroporto(String tipo) {

        if (tipo.equals("carro")) {
            System.out.println("Indo de carro");
        } 
        else if (tipo.equals("onibus")) {
            System.out.println("Indo de ônibus");
        } 
        else if (tipo.equals("bicicleta")) {
            System.out.println("Indo de bicicleta");
        }
    }
}

Problemas:

  • Muitos if/else
  • Difícil de manter
  • Não segue Open/Closed

🛠️ Como implementar o Strategy

Passo a passo para aplicar o padrão Strategy na prática:
  1. 1. Identifique o problema
    Procure um algoritmo que muda com frequência ou um código cheio de if/else. 
    2. if (tipo.equals("carro")) { ... }
else if (tipo.equals("onibus")) { ... }
else if (tipo.equals("bicicleta")) { ... }
  2. 2. Crie a interface Strategy
    Defina um contrato comum para todos os comportamentos. 
    3. interface TransporteStrategy {
    void ir();
}
  3. 3. Separe as implementações
    Cada comportamento vira uma classe diferente. 
    4. class Carro implements TransporteStrategy {
    public void ir() {
        System.out.println("Indo de carro");
    }
}
  4. 4. Crie o Contexto
    O contexto usa a estratégia sem saber detalhes dela. 
    5. class Contexto {

    private TransporteStrategy strategy;

    public void setStrategy(TransporteStrategy strategy) {
        this.strategy = strategy;
    }

    public void executar() {
        strategy.ir();
    }
}
  5. 5. O cliente escolhe a estratégia
    Quem usa o sistema decide qual comportamento usar. 
    6. public class Main {
    public static void main(String[] args) {

        Contexto contexto = new Contexto();

        contexto.setStrategy(new Carro());
        contexto.executar();

        contexto.setStrategy(new Onibus());
        contexto.executar();
    }
}

💡 O que está acontecendo aqui?

  • O comportamento foi extraído da classe principal
  • Cada algoritmo virou uma classe independente
  • O contexto apenas delegada a execução
  • O sistema fica aberto para extensão e fechado para modificação

🏗️ Strategy na Arquitetura Hexagonal

Na Arquitetura Hexagonal (Ports and Adapters), o padrão Strategy se encaixa principalmente dentro da camada de domínio ou casos de uso, dependendo da responsabilidade do comportamento.

📍 Localização ideal

  • Interface Strategy: fica no Domínio (regra de negócio)
  • Implementações: podem ficar:
    • no Domínio (se forem regras puras)
    • ou em Adaptadores (se dependem de tecnologia externa)
  • Contexto: geralmente fica nos Casos de Uso

Isso garante que o domínio continue independente de frameworks, banco de dados ou detalhes externos.

💡 Por que funciona tão bem na Hexagonal?

A Arquitetura Hexagonal promove o desacoplamento entre o núcleo da aplicação e o mundo externo. O padrão Strategy segue exatamente esse princípio, pois também desacopla o comportamento de quem o utiliza.

Enquanto a Hexagonal separa "o que o sistema faz" de "como ele se conecta", o Strategy separa "o que deve ser feito" de "como é feito".

🏢 Arquitetura em Camadas

  • Strategy pode ser usado, mas:
    • fica preso na camada de serviço
    • tende a gerar acoplamento entre camadas
    • difícil separar regras de negócio puras

❌ Problema: a lógica ainda depende da estrutura da aplicação

🧼 Clean Architecture

  • Strategy também se encaixa bem
  • fica nas Use Cases ou Entities
  • segue inversão de dependência

✅ Funciona muito bem, mas é mais abstrata e complexa de implementar

🔷 Arquitetura Hexagonal

  • Strategy se encaixa naturalmente como comportamento plugável
  • combina perfeitamente com o conceito de Ports
  • permite trocar implementações sem afetar o domínio

✅ Melhor alinhamento conceitual e prático

🏆 Conclusão

O padrão Strategy se encaixa melhor na Arquitetura Hexagonal porque ambos compartilham o mesmo princípio fundamental: desacoplamento.

Na arquitetura em camadas, o Strategy tende a ficar limitado e acoplado à estrutura. Na Clean Architecture, ele se encaixa bem, mas com maior complexidade estrutural.

Já na Arquitetura Hexagonal, o Strategy atua de forma natural como um componente intercambiável dentro do domínio ou dos casos de uso, permitindo a substituição de comportamentos sem impactar o núcleo da aplicação.

Isso torna o sistema mais flexível, testável e preparado para mudanças, atendendo perfeitamente aos princípios de design moderno como SOLID e baixo acoplamento.
